# BaneTask 2.1.0：把量子化学工作流从“生成输入”推进到“可查询结果台账”

现在的 BaneTask 已经不是一个“只帮你拼输入文件”的小工具，而是一个把**任务描述、依赖串联、命令脚本生成、结果快照、SQLite 派生库查询**串在一起的量子化学工作流生成器。

## 它现在解决什么问题

如果你经常做量子化学计算，大概率会反复碰到这些事：

- 同一批结构要反复生成 Gaussian / ORCA / 其他程序输入
- 上游任务一变，下游起点、guess、脚本都得跟着改
- 计算完成后，日志、fchk、分析结果散落在各个目录里
- 批量项目里想回答“哪个分子过了 TS 判据”“哪一组能量差更小”时，往往还得自己 grep / 手工抄表

BaneTask 的定位一直没变：**用一个简洁的 `.bt` 文件描述工作流，让重复劳动自动化**。

但 2.1.0 这版的重点，是把“自动化生成”进一步推到了“结果可查询”。

## 2.1.0 现在有的核心能力

### 1. 统一的 `.bt` 工作流描述

当前代码内置支持：

- `gaussian`
- `orca`
- `script`
- `other`

你可以在同一个 `.bt` 里混用它们：

```text
$opt
  %source origin
  %program gaussian
  %keywords "opt freq B3LYP/6-31G(d)"

$sp
  %source opt
  %program orca
  %keywords "wB97M-V def2-TZVP def2/J RIJCOSX TightSCF"
```

变量系统、`#INCLUDE` / `&INCLUDE`、`extra.yaml` / `extra.bt` 这些机制也都还在，因此把“公共模板”和“目录私货”拆开维护会比以前舒服很多。

### 2. 依赖不只是“等上游跑完”，还可以做条件分支

当前 `%control` 已经支持：

- `totcore`
- `totmem`
- `runifdef`
- `runif`
- `verbosity`

`runif` 是这一版非常实用的点。它允许你直接基于上游结果快照决定下游任务要不要生成，例如：

```text
$ts_refine
  %source ts_guess
  %control
    runif {{ nimag }} = 1
    runif {{ imag1 }} < -500
  %keywords "opt=(ts,calcfc) B3LYP/6-31G(d)"
```

这意味着 BaneTask 不再只是“按顺序排队”，而是能做一点真正有意义的 workflow gating。

### 3. 每个任务都能留下可复用的结果快照

当前命令链里，只要你启用了结果记录，任务完成后会额外生成：

```text
任务目录/.bane.result.kv
Results/DB/snapshots/<run_id>.kv
Results/DB/records/<run_id>.json
Results/DB/queue/<run_id>.ready
```

`.bane.result.kv` 是给 `runif` 和本地快速查看用的；`records/*.json` 则是给后面的 SQLite 派生库 ingest 用的。

也就是说，现在一个任务跑完，结果不只是“留一个 log 在那儿”，而是会多一份结构化摘要，后面可以被程序继续消费。

### 4. 现在真的有 SQLite 结果库了

2.1.0 里一个非常实际的变化，是 banedb 不再停留在“蓝图”上。

当前代码已经有：

```bash
banetask db init|sync|rebuild|query|export
btask db init|sync|rebuild|query|export
```

默认数据库位置是：

```text
Results/DB/project.db
```

常用视图是：

```sql
SELECT case_key, task_name, energy, nimag
FROM task_flat_latest;
```

这里的 `task_flat_latest` 会自动帮你把每个 `case + task_name` 的最新记录平铺出来，用来做筛选、导出和后续比较都很顺手。

### 5. 项目模式现在可以直接比较两个任务块

如果你是 `.projbt` 项目流，当前代码还支持：

```bash
btask compare opt td --metric energy --path ./my_project
btask compare td td_t1 --metric tddft.total_energy --path ./my_project
```

这会按照 `project_key + case_key` 对齐左右两边的 latest 结果，直接给出 `left_value`、`right_value` 和 `delta`。

以前很多“批量项目横向比较”的事，都是自己导表后再算；现在至少已经有一个够实用的原生入口了。

### 6. Windows 不再只是顺手兼容

当前代码的脚本输出已经明显考虑了双平台：

- Linux / macOS 默认输出 bash 风格脚本
- Windows 默认输出 cmd / bat 风格脚本
- 也支持切到 Git Bash 风格

另外 Windows 目标下还会构建 `btc.exe`，用于从 `.btc` 模板快速生成 `.bt`，并可选直接调用 `run.bat`。

这不是“所有生态都已经完美双平台”——例如很多个人工具链命令显然还是更偏 bash——但至少 BaneTask 自己的输入 / 脚本生成层面，已经不再把 Windows 视为完全旁路的存在。

## 一个更像“当前代码面貌”的例子

```text
autorun: false

define:
  func: B3LYP
  basis: 6-31G(d)

$opt
  %source origin
  %program gaussian
  %control
    totcore 16
    totmem 64000
  %keywords "opt freq [func]/[basis]"
  %process
    scripts fchk

$sp
  %source opt
  %program orca
  %control
    totcore 32
    totmem 96000
  %keywords "wB97M-V def2-TZVP def2/J RIJCOSX TightSCF"

$ts_check
  %source opt
  %control
    runif {{ nimag }} = 1
    runif {{ imag1 }} < -500
  %keywords "freq [func]/[basis]"
```

这个例子里：

- `opt` 做 Gaussian 优化 + 频率
- `sp` 取 `opt` 结构跑 ORCA 单点
- `ts_check` 只有在上游结果快照显示 `nimag == 1` 且首个虚频足够大时才会生成

这比“所有任务无脑一路铺开”更接近真实项目里想要的行为。

## 构建层面需要知道的现实情况

当前源码构建至少需要：

- CMake >= 3.10
- C++17 编译器
- SQLite3 开发文件
- `bane::geometry` target（来自安装好的 `bane_lib` 或同级 `../banelib`）

换句话说，现在 README 里如果还把它写成一个“纯粹的单文件小工具”，就已经不准确了。结果库和 geometry 后端已经是正式依赖的一部分。

同时，也要诚实一点：像 `banejs`、`ofakeg`、`Multiwfn`、`banewfn` 这些运行时工具，仍然需要你自己准备。BaneTask 会调它们，但不会神奇地替你装好整条工具链。

## 这版最值得记住的，不是“功能更多了”

我觉得 2.1.0 最关键的变化，不是某一个单点命令，而是 BaneTask 的角色更清晰了：

- 前端是 `.bt` 任务描述语言
- 中间是输入 / command / workflow 生成
- 后端是结果快照、记录 JSON 与 SQLite 派生库

这让它从“量子化学输入生成器”更像一个轻量 workflow workbench：

- 能生成
- 能组织
- 能判断
- 能查询
- 还能在项目级做简单比较

这一步很重要，因为它决定了以后扩展时，不用总回到“grep 日志、脚本堆命令”的老路上。

## 适合谁用

到当前这版，BaneTask 比较适合：

- 已经有自己算化脚本生态，但想把流程描述统一起来的人
- 一批分子、一组模板、很多相似计算要反复铺的人
- 需要用 `runif` 做简单条件分支的人
- 想把项目结果变成可查询台账，而不是只留一堆目录的人
